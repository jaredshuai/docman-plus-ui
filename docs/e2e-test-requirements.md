# E2E 测试要求

## 1. 目标

本要求用于规范 `docman-plus-ui` 与 `docman-ruoyi` 联调后的端到端测试范围、执行前提、通过标准和交付物。

E2E 测试的核心目标不是替代单元测试，而是验证：

- 用户从浏览器入口到后端接口的完整业务链路可用
- 关键页面在真实登录态下可正常渲染和交互
- 核心接口返回业务成功数据，而不是仅 HTTP 200
- 页面跳转、权限控制、表单提交、文件上传下载等主流程不回归

## 2. 适用范围

本轮 E2E 覆盖以下范围：

- 登录与基础导航
- `docman` 菜单与页面可达性
- `docman` 仪表盘
- 项目管理
- 文档中心
- 流程编排
- 归档管理
- 插件列表与插件执行日志

本轮不强制覆盖以下范围：

- 全系统所有菜单
- 三方登录
- 监控中心
- 压力测试
- 多浏览器兼容性矩阵
- 移动端适配

## 3. 推荐工具

推荐使用 `Playwright`，原因：

- 对登录态、路由跳转、接口拦截、文件上传下载支持完整
- 与当前前端技术栈兼容性高
- 能输出 trace、截图、视频，便于排查回归问题

最低要求：

- Node.js `>= 20`
- Chromium 作为首个必测浏览器

## 4. 测试环境要求

E2E 默认运行在本机 Docker 联调环境。

要求如下：

- 前端入口：`http://localhost:18080`
- 后端网关入口：`http://localhost:18080/prod-api`
- 后端直连：`http://localhost:18081`
- 当前部署目录：`D:/codespace/docman-ruoyi/script/docker/docman-local`

执行前必须满足：

- Docker 容器全部启动成功
- `mysql`、`redis`、`ruoyi-admin`、`nginx-web` 状态正常
- `/auth/code` 返回业务成功
- `admin / admin123` 可登录

如果测试环境使用本机 Docker，要求先执行：

1. 前端打包
   `cd D:/codespace/docman-plus-ui`
   `npm run build:prod`

2. 后端打包
   `cd D:/codespace/docman-ruoyi`
   `mvn clean package -pl ruoyi-admin,ruoyi-extend/ruoyi-snailjob-server -am -Pprod -DskipTests`

3. 启动联调环境
   `cd D:/codespace/docman-ruoyi/script/docker/docman-local`
   `docker compose up -d --no-build`

## 5. 测试账号要求

默认账号：

- 用户名：`admin`
- 密码：`admin123`

要求：

- E2E 必须显式校验登录成功后的跳转结果
- 不允许依赖浏览器残留登录态
- 每条用例应在 `beforeEach` 或全局 setup 中建立可控登录态

## 6. 测试数据要求

### 6.1 基础原则

- 测试应尽量使用可重复初始化的数据
- 除“只读 smoke”外，新增/修改/删除类用例必须自己准备测试数据
- 不允许依赖人工临时造数
- 每条会修改数据的用例都必须具备清理策略

### 6.2 当前建议

本机 Docker 环境允许保留一组演示数据用于 smoke：

- 演示项目
- 2 条文档记录
- 1 条节点截止记录
- 2 条插件执行日志

但正式 E2E 要求：

- 创建类用例：使用唯一名称，例如 `e2e-project-时间戳`
- 删除类用例：删除自己创建的数据
- 查询类用例：不依赖固定排序之外的脏数据

## 7. 必测场景

### 7.1 P0 冒烟

以下场景每次发布前必须通过：

1. 登录页打开成功
2. 使用 `admin / admin123` 登录成功
3. 首页加载成功
4. `文档管理` 一级菜单可见
5. `仪表盘` 页面打开成功
6. `项目管理` 页面打开成功
7. `文档中心` 页面打开成功
8. `流程编排` 页面打开成功
9. `归档管理` 页面打开成功
10. `插件列表` 页面打开成功

### 7.2 仪表盘

要求验证：

- 页面无白屏
- 四个统计卡片正常渲染
- 项目进度表正常渲染
- 即将超期节点表正常渲染
- 插件执行统计表正常渲染
- 页面接口返回业务成功，不允许仅依赖 HTTP 200

至少校验接口：

- `/docman/dashboard/overview`
- `/docman/dashboard/project-progress`
- `/docman/dashboard/deadline-alert`
- `/docman/dashboard/plugin-stats`

### 7.3 项目管理

要求覆盖：

1. 打开项目列表
2. 新增项目
3. 查询新增结果
4. 编辑项目名称或备注
5. 删除测试项目

断言要求：

- 新增后表格中可见目标项目
- 编辑后字段更新正确
- 删除后列表中不可见

### 7.4 文档中心

要求覆盖：

1. 打开文档列表
2. 切换项目后列表可刷新
3. 上传允许类型文件成功
4. 不允许类型文件被拦截
5. 超大文件被拦截

若环境允许下载验证，则继续覆盖：

6. 下载文档成功
7. 删除文档成功

### 7.5 流程编排

要求覆盖：

1. 页面可打开
2. 流程定义列表可加载
3. 绑定流程成功
4. 启动流程成功

如果环境中没有现成流程定义，则本条可降级为：

- 校验定义列表接口正常返回
- 页面无报错

### 7.6 归档管理

要求覆盖：

1. 页面可打开
2. 归档历史可查询
3. 对无可归档数据的提示正确

如测试环境具备完整文档与流程数据，可进一步覆盖：

4. 发起归档
5. 查询归档结果

### 7.7 插件列表

要求覆盖：

1. 插件列表可展示
2. 插件执行日志列表可展示
3. 切换项目后日志可刷新

如环境具备可执行插件配置，可进一步覆盖：

4. 手动触发插件执行
5. 执行结果可在日志中看到

## 8. 断言要求

### 8.1 页面级

每个页面至少满足：

- 关键标题可见
- 主表格或主内容区域可见
- 无全局错误弹窗
- 无白屏

### 8.2 接口级

对关键接口不能只断言 HTTP 状态码。

必须同时断言：

- HTTP 状态为 `200`
- 返回体 `code === 200`
- 返回数据结构符合页面预期

### 8.3 日志级

测试执行过程中不允许出现：

- 后端 `500`
- SQL 异常
- 页面路由不存在
- 未处理 Promise 异常

浏览器控制台允许忽略的项必须白名单化，不允许“默认忽略所有 error”。

## 9. 稳定性要求

E2E 用例必须满足：

- 可重复执行
- 顺序无依赖
- 单条失败不污染后续环境
- 不依赖人工点击验证码或人工造数

对于不稳定点，要求显式处理：

- 等待接口完成后再断言
- 对表格异步加载使用明确等待条件
- 对下载、上传、弹窗使用稳定选择器

## 10. 选择器要求

前端页面必须优先提供稳定选择器。

建议顺序：

1. `data-testid`
2. 稳定文本
3. 稳定表单 label
4. 稳定路由 URL

禁止长期依赖：

- 动态生成 class
- 深层 CSS 路径
- 随版本变化明显的组件内部 DOM 结构

## 11. 失败产物要求

每次 E2E 执行失败时，至少保留：

- 失败截图
- 浏览器 trace
- 控制台日志
- 关键网络请求记录

## 12. 通过标准

发布前最低通过标准：

- P0 冒烟用例全部通过
- 不允许出现阻塞登录、菜单加载、页面白屏、接口 500、核心 CRUD 失败
- `docman` 关键页面无业务异常返回

建议通过标准：

- P0 全通过
- P1 主流程通过率 `>= 95%`
- 同一条用例重跑 3 次无随机失败

## 13. 后续实施建议

建议按两阶段实施：

### 阶段一

- 先补 `Playwright` 基础设施
- 先做 `P0` 冒烟
- 先做登录、仪表盘、项目管理列表页

### 阶段二

- 补 CRUD
- 补上传下载
- 补流程与归档
- 补插件触发

## 14. 当前结论

基于当前代码和本机 Docker 环境，E2E 的第一批目标建议是：

- 登录
- 仪表盘
- 项目管理 CRUD
- 文档中心上传
- 插件列表查询

这批最能覆盖真实联调价值，也最适合先落地。
